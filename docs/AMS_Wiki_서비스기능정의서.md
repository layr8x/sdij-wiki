## 1. 프로젝트 개요

### 1.1 추진 배경

AMS(Academy Management System)는 시대인재의 재종·단과 운영을 위한 핵심 시스템으로, 회원·강좌·청구·결제·환불 등 대부분의 운영 업무가 이 시스템을 통해 수행된다. 현재 기능 가이드와 운영 기준이 모듈별로 분산 관리되고 있으며, PDF 변환 후 카카오톡으로 공유하는 방식이 문서 관리·업데이트 추적·보안 측면에서 구조적 한계를 노출하고 있다.

| 문제 영역 | 현황 | 영향 |
|---|---|---|
| 문서 관리 | PDF 변환 후 카카오톡 공유 | 레이아웃 깨짐, 최신 버전 확인 불가, 변경 이력 관리 불가 |
| 기능 업데이트 관리 | 카카오톡 단발성 공지 | 가이드 미반영, 이전 업데이트 이력 확인 불가, 변경 기능 즉시 확인 불가 |
| 보안·정보 관리 | 외부 PDF 파일 공유 중심 | 내부 문서 외부 유출 가능성, 접근 권한·열람 이력 관리 불가 |

### 1.2 AS-IS / TO-BE

| 구분 | AS-IS (현재) | TO-BE (개선) |
|---|---|---|
| 가이드 관리 | 담당자 개별 문서 작성 → PDF 공유 | 위키 기반 단일 구조 통합 관리, 변경 이력 자동 추적 |
| 기능 업데이트 | 카카오톡 공지로 전달, 이전 변경 이력 확인 불가 | 위키 가이드 + 업데이트 이력 동시 관리, 변경 알림 자동화 |
| 정보 탐색 | 카톡·담당자 문의로 수동 확인, 탐색 시간 5~10분 | 위키 검색으로 30초 내 즉시 확인, 동의어 사전·최근 조회 지원 |
| 버전 관리 | 최신 문서 여부 확인 불가 | 문서 버전·변경 이력 체계적 관리 |
| 접근·보안 | PDF 외부 공유 가능, 열람 이력 관리 불가 | 내부 위키 기반 접근 권한 관리, 로그 기반 열람 추적 |

### 1.3 추진 방향

AMS 운영 정보를 위키 기반으로 통합 관리하는 체계를 구축한다. 핵심 추진 방향은 세 가지이다.

| 추진 방향 | 설명 |
|---|---|
| 운영 가이드 통합 관리 | 모듈별 분산 가이드를 위키 기반 단일 구조로 통합 |
| 기능 업데이트 기록 관리 | 기능 개선·정책 변경 시 위키 내 업데이트 기록 체계 구축 |
| 운영 지식 공유 기반 마련 | 주요 운영 사례 및 FAQ를 정리하여 운영 지식 자산화 |

## 2. 글로벌 UX/UI 레퍼런스 벤치마크

AMS 위키의 UX 설계에 직접적으로 참고할 수 있는 글로벌 서비스 7개를 선정하고, 각 서비스의 레이아웃·네비게이션·검색·콘텐츠 구조를 분석하였다. 선정 기준은 내부 운영 도구(Knowledge Base / Help Center / Documentation) 용도의 서비스로서, PC 환경에서 높은 사용성을 입증한 사례이다.

### 2.1 레퍼런스 서비스 종합 비교

| 서비스 | 레이아웃 | 네비게이션 | 검색 패턴 | 핵심 UX 강점 |
|---|---|---|---|---|
| **Stripe Docs** | 3-column (사이드바+본문+코드) | 좌측 트리 + 우측 미니맵, 스크롤 연동 하이라이트 | Cmd+K 글로벌 검색, 자동완성 + 스니펫 미리보기 | 코드 예시와 설명의 맥락 동기화, 호버 시 코드 하이라이트 연동 |
| **Notion Help** | 2-column (사이드바+본문) | 카테고리 그리드 + 검색, 아이콘 기반 카테고리 카드 | 실시간 검색 + 카테고리 필터, 최근 인기 문서 노출 | 시각적 카테고리 랜딩, 관련 문서 자동 추천 |
| **Zendesk Guide** | 2-column (카테고리+본문) | 카테고리 > 섹션 > 아티클, 브레드크럼 네비게이션 | 검색바 히어로 영역, 결과 페이지 카테고리 패싯 | Help Center 특화 템플릿, SEO 최적화 구조 |
| **GitBook** | 3-column (사이드바+본문+미니맵) | 좌측 트리 + 우측 On-this-page, 그룹 접기/펼치기 | Cmd+K 커맨드 팔레트, AI 요약 답변 + 출처 링크 | AI 기반 문서 검색 답변, 마크다운 네이티브 작성 |
| **Intercom Help** | 2-column (카테고리+본문) | 홈: 카테고리 카드 그리드, 아티클: 사이드 내비게이션 | 자연어 검색, Instant Answer Bot 연동 | 메신저 위젯 통합, 검색→봇→휴먼 에스컬레이션 |
| **Atlassian Docs** | 2-column (좌측 트리+본문) | 스페이스 > 페이지 트리, 즐겨찾기·최근 보기 섹션 | Quick Search + CQL, 스페이스 범위 필터 | 실시간 협업 편집, 인라인 코멘트·멘션 |
| **Slack Help** | 2-column (카테고리+본문) | 대카테고리 그리드, 아티클 내 스텝 바이 스텝 | 검색바 상단 고정, 자동완성 + 인기 토픽 추천 | 스크린샷 기반 단계별 가이드, 플랫폼별 탭 분기 |

### 2.2 AMS 위키 적용 전략: 레퍼런스별 차용 요소

각 글로벌 레퍼런스에서 AMS 위키에 직접 적용할 패턴을 선별하였다. 단순 벤치마크가 아니라, 시대인재 운영자(상담실장)의 사용 맥락에 맞춰 변형·적용하는 전략이다.

| 레퍼런스 | 차용 패턴 | AMS 위키 적용 방법 | 적용 근거 |
|---|---|---|---|
| Stripe Docs | 3-column 레이아웃, 우측 미니맵 스크롤 연동 | 좌측 사이드바(260px) + 본문(max 840px) + 우측 On-this-page(200px), 스크롤 시 현재 섹션 하이라이트 | 1920px 해상도에서 3-column 최적, 가이드 내 섹션 탐색 속도 향상 |
| Stripe Docs | Cmd+K 글로벌 검색 | '/' 단축키 글로벌 검색 오버레이, 자동완성 + 모듈 뱃지 + 스니펫 | 상담실장 30초 내 답 도달 원칙, 키보드 기반 파워유저 효율 |
| Notion Help | 아이콘 기반 카테고리 카드 그리드 | 위키 홈: 모듈별 아이콘+제목+가이드수 카드 그리드 (3×3, 1920px 기준) | 첫 방문자 탐색 진입점, 모듈 구조 즉시 파악 |
| Zendesk Guide | 검색바 히어로 영역, 카테고리 패싯 필터 | 위키 홈 상단: 검색바 히어로 (h=120px), 검색 결과: 모듈 필터 칩 | 60% 사용자가 검색으로 진입, 검색 가시성 극대화 |
| GitBook | AI 기반 문서 검색 답변 | 4단계(9~12M) AI 챗봇에서 검색 결과 상단 AI 요약 답변 카드 | FAQ 질문의 반자동 답변, 위키 콘텐츠 RAG 소스 |
| Intercom | 검색→봇→휴먼 에스컬레이션 | 검색 미발견 시 '개선/불편사항 접수' 링크, 4단계 AI 챗봇 에스컬레이션 룰 | CS 감소 목표와 직결, 자동화→휴먼 전환 시점 설계 |
| Atlassian | 즐겨찾기·최근 보기 섹션 | 사이드바 상단: 최근 조회 가이드 5개 + 즐겨찾기(북마크) 기능 | 운영자 반복 참조 패턴 지원, Known-item 검색 60% 대응 |
| Slack Help | 스크린샷 기반 단계별 가이드, 플랫폼별 탭 분기 | 사용 방법 블록: 스텝퍼 + 인라인 스크린샷, 대상별 탭(운영자/실장/관리자) | 실제 AMS 화면 맥락 보존, 역할별 정보 필터링 |

> **Reference** — Stripe의 3-column 레이아웃은 API 문서에서 시작되어 Knowledge Base 분야의 사실상 표준이 되었다(Moesif Blog, 2023). 좌측 네비게이션, 중앙 콘텐츠, 우측 코드/미니맵 구조는 1920px+ 해상도에서 빈 공간을 효과적으로 활용하면서도, 사용자의 시선 이동 거리를 최소화한다.

## 3. UX 설계 원칙 (Design Principles)

AMS 위키의 모든 UI/UX 의사결정은 다음 6개 원칙을 기준으로 판단한다. 원칙 간 충돌 시 상위 번호 원칙이 우선하며, 각 원칙에는 글로벌 레퍼런스 근거를 명시한다.

| # | 원칙명 | 정의 | 글로벌 레퍼런스 | AMS 적용 |
|---|---|---|---|---|
| 1 | **30초 내 답 (Speed-to-Answer)** | CS 응대 중 30초 이내에 필요한 정보에 도달 | Stripe: Cmd+K 검색 → 즉시 결과 / Zendesk: 히어로 검색바 최우선 배치 | '/' 단축키 검색, 최근 조회 5개 즉시 노출, 자동완성 300ms 디바운스 |
| 2 | **맥락 보존 (Context Continuity)** | AMS 실제 화면과 위키 가이드 사이 맥락 전환 비용 최소화 | Stripe: 설명 호버 시 코드 하이라이트 / Slack: 스크린샷 기반 스텝 가이드 | AMS 메뉴 경로 딥링크, 인라인 스크린샷, 단계별 체크리스트 |
| 3 | **점진적 노출 (Progressive Disclosure)** | 핵심 정보 선행 노출, 상세는 사용자 요청 시 확장 | Notion: 토글 블록 기본 접힘 / GitBook: 섹션별 접기/펼치기 | 운영 케이스 아코디언, TL;DR 상단 고정, 유의사항 접기/펼치기 |
| 4 | **신뢰 표지 (Trust Signals)** | 콘텐츠 최신성·정확성·권위를 UI 레벨에서 즉시 인지 | Atlassian: 버전 이력·수정일·작성자 / Zendesk: '이 문서가 도움됨?' 피드백 | 업데이트 뱃지(7일간), 버전+수정일+작성자, 유용성 피드백 위젯 |
| 5 | **제로 온보딩 (Zero Onboarding)** | 별도 교육 없이 위키 자체가 사용법을 설명 | Notion: 빈 상태 가이드 텍스트 / Intercom: 카테고리 카드 그리드 | 빈 상태(Empty State) 가이드, 홈 카테고리 카드 그리드, 작성 툴 인라인 힌트 |
| 6 | **F-패턴 최적화 (Scannable Layout)** | 상단 2줄에 핵심 요약, 좌측에 라벨·번호 고정 | NNGroup F-패턴 연구 / UXCC: 헤딩+좌측 콘텐츠 프론트로딩 | TL;DR 카드 상단 2줄, 스텝 번호 좌측 고정, 케이스 라벨 좌측 정렬 |

## 4. 사용자 분석 및 여정 맵

### 4.1 사용자 세그먼트 정의

AMS 위키의 사용자는 역할·권한·정보 소비 패턴이 서로 다른 3개 세그먼트로 구분된다. 각 세그먼트의 정보 요구 수준과 사용 빈도가 상이하므로, 위키 IA와 탐색 UX는 이를 반영해야 한다.

| 세그먼트 | 역할 | 주요 태스크 | 정보 요구 수준 | 사용 빈도 | 사용 환경 |
|---|---|---|---|---|---|
| **운영자 (상담실장)** | 학부모/학생 대면 상담, CS 1차 대응 | CS 응대 중 SOP 즉시 참조, 회원 이관/통합 처리, 출결 이슈 해결 | 실행 절차 중심 (How-to) | 매일 10~20회 (CS 발생 시) | PC 1920×1080 + 모바일 병행 |
| 실장 (캠퍼스 실장) | 캠퍼스 운영 총괄, 정책 판단·승인 | 환불 승인 기준 확인, 정책 변경 사항 파악, 이슈 에스컬레이션 판단 | 판단 기준 중심 (When/Why) | 주 3~5회 (판단 필요 시) | PC 위주, 간헐적 모바일 |
| 관리자 (본사 기획자) | 시스템 기획·운영 정책 수립, 가이드 작성·관리 | 가이드 신규 작성/수정, 업데이트 이력 관리, 콘텐츠 커버리지 모니터링 | 전체 구조 이해 (What/How/Why) | 주 5~10회 (작성·관리) | PC 전용 |

> **UX Rationale** — 가장 높은 사용 빈도는 운영자(상담실장)이며, 'CS 응대 중 30초 내 답을 찾아야 하는' 시간 압박이 핵심 제약 조건이다. 1920×1080 PC 환경에서 검색바와 사이드바를 동시 노출할 수 있는 3-column 레이아웃이 이 제약을 해소한다.

### 4.2 핵심 사용자 여정 맵: 운영자(상담실장) CS 대응 흐름

상담실장이 CS 대응 중 AMS 위키를 사용하는 전형적 여정을 5단계로 정의한다. 각 단계의 페인 포인트가 곧 UX 설계 요구사항이다.

| 단계 | 사용자 행동 | 페인 포인트 | UX 해결 전략 | 글로벌 레퍼런스 |
|---|---|---|---|---|
| 1. 트리거 | CS 인입 (카카오톡/전화/대면) | 어디서 정보를 찾는지 모름, 검색 진입점 불명확 | 히어로 검색바 최상단 배치, '/' 단축키 접근 | Zendesk 히어로, Stripe Cmd+K |
| 2. 탐색 | 검색어 입력 또는 메뉴 탐색 | 학부모 표현 ≠ AMS 용어 (예: '돈 돌려받기' ≠ '환불') | 동의어 사전 매핑, 최근 조회 바로가기, 모듈 뱃지 필터 | Zendesk 패싯, Atlassian 최근보기 |
| 3. 확인 | 가이드 스캔, 해당 케이스 매칭 | 긴 문서에서 내 케이스 찾기 어려움, 스크롤 과다 | TL;DR 상단 고정, 운영 케이스 아코디언, 우측 미니맵 스크롤 연동 | Stripe 미니맵, Notion 토글블록 |
| 4. 실행 | AMS에서 실제 처리 (이관/환불/출결) | 가이드↔AMS 화면 전환 빈번, 절차 중간 놓침 | AMS 메뉴 경로 딥링크, 스크린샷 인라인, 단계별 체크리스트 | Slack 스텝 가이드, Stripe 코드 동기화 |
| 5. 완료 | 처리 완료, 동료에게 공유 | URL 공유 시 미리보기 부재, 어떤 가이드인지 모름 | OG 메타태그 적용, 가이드 제목+모듈+수정일 프리뷰 노출 | 모든 서비스 공통 |

### 4.3 사용자 여정 맵: 관리자(기획자) — 가이드 작성 흐름

| 단계 | 사용자 행동 | 페인 포인트 | UX 요구사항 |
|---|---|---|---|
| 1. 기획 | 작성할 가이드 범위 결정, 기존 중복 확인 | 어떤 가이드가 이미 있는지 파악 어려움 | 가이드 인벤토리 대시보드, 모듈별 커버리지 시각화 |
| 2. 작성 | 입력 폼에 내용 채움, 스크린샷 첨부 | 긴 폼 작성 중 중간 이탈/유실, 이미지 첨부 불편 | 섹션별 자동 임시 저장, 드래그앤드롭 이미지 업로드 |
| 3. 검수 | 미리보기 확인, 동료 리뷰 요청 | 미리보기↔편집 전환 불편, 리뷰 코멘트 수렴 분산 | Split-view 편집/미리보기, 인라인 코멘트 기능 |
| 4. 발행 | 상태 전환 (작성중→배포완료), URL 확인 | 발행 후 실수 발견 시 롤백 불안 | 발행 전 최종 확인 모달, 1-click 롤백 버튼 |
| 5. 유지 | 업데이트 필요 시 수정 발행, 조회수·피드백 모니터링 | 어떤 가이드가 오래된 것인지 모름 | 스테일 콘텐츠 알림(90일 미수정), 가이드 하단 유용성 피드백 위젯 |

## 5. PC 1920×1080 레이아웃 사양

메인 타겟 해상도 1920×1080에서의 레이아웃을 px 단위로 정의한다. 전체 너비 1920px 중 좌우 여백 80px씩을 제외한 컨텐츠 영역 1760px을 3-column으로 배분한다.

### 5.1 전체 그리드 시스템

| 영역 | 너비(px) | 위치 | 콘텐츠 | 동작 |
|---|---|---|---|---|
| **글로벌 헤더** | 1920 (full) | top, h=56px | 로고 + 글로벌 검색바(w=480px) + 프로필 | 스크롤 시 sticky 고정, 검색바: '/' 키로 포커스 |
| **좌측 사이드바** | 260 | left, x=80 | 최근 조회(5개) + 즐겨찾기(북마크) + 모듈 트리 네비게이션 | sticky(top=56px), 현재 위치 하이라이트, 4px 좌측 액센트 바 |
| **본문 영역** | 840 (max, flex) | center | 가이드 콘텐츠 10개 블록 | 좌우 패딩 40px, 라인 길이 max 80자, Scroll spy 연동 |
| **우측 미니맵** | 200 | right, x=1580 | On-this-page 목차, 현재 섹션 하이라이트, '맨 위로' 링크 | sticky(top=56px), IntersectionObserver 스크롤 위치 연동 |
| **좌우 여백** | 80 × 2 = 160 | 양쪽 끝 | 빈 공간 | 1440px 이하 시 40px로 축소 |

> **Reference** — Stripe Docs의 3-column 레이아웃은 sidebar=280px, content=auto, right-panel=200~240px 구조이다. AMS 위키는 코드 패널 대신 On-this-page 미니맵으로 변환하여 운영 가이드 맥락에 맞춘다. GitBook도 동일한 구조를 채택하고 있다.

### 5.2 해상도별 반응형 브레이크포인트

| 브레이크포인트 | 해상도 | 레이아웃 변화 | 비고 |
|---|---|---|---|
| **XL (메인 타겟)** | ≥1920px | 3-column: 사이드바 260 + 본문 840 + 미니맵 200, 좌우 여백 80px | 최적 경험 |
| L | 1440~1919px | 3-column 유지, 여백 40px 축소, 본문 영역 flex 축소 | 지원 |
| M | 1280~1439px | 2-column: 미니맵 비노출, 사이드바 240px + 본문 flex | 지원 |
| S | 1024~1279px | 2-column: 사이드바 접힘(햄버거), 본문 풀 너비 | 최소 지원 |
| XS (모바일 참조) | <1024px | 1-column: 하단시트 네비게이션, 검색 풀스크린 | 조회 전용 |

| **검색 진입점** | h=44px | '/' + '가이드 검색' 텍스트, border: 1px solid #E0E0E0 | 클릭 시 글로벌 검색 오버레이 활성화 |

## 6. 위키 홈 페이지 UI 설계

위키 홈은 사용자가 위키에 최초 진입하거나 검색 이외의 탐색을 시작하는 랜딩 페이지이다. Notion Help의 카테고리 카드 그리드와 Zendesk Guide의 히어로 검색바를 결합한 하이브리드 구조를 채택한다.

### 6.1 홈 페이지 구성 블록 (1920×1080 기준)

| # | 블록 | 높이(px) | 콘텐츠 | UX 근거 |
|---|---|---|---|---|
| 1 | **히어로 검색 영역** | h=160 | 제목: 'AMS 운영 가이드', 검색바: w=640, h=44, border-radius=22px, 하단: 인기 검색어 태그 5개 | Zendesk Guide 패턴, 검색 진입 유도 최우선 |
| 2 | **모듈 카테고리 카드 그리드** | h=auto | 3열 × N행 카드 그리드, 카드: 아이콘 + 모듈명 + 가이드 수 + 1줄 설명, 카드 크기: w=340, h=120, gap=24 | Notion Help 패턴, 모듈 구조 즉시 파악, 첫 방문자 온보딩 |
| 3 | **최근 업데이트 가이드** | h=auto (max 5행) | 제목: '최근 업데이트된 가이드', 리스트: 모듈뱃지 + 가이드제목 + 수정일 + '업데이트됨' 뱃지, 항목 높이: h=48 | 신뢰 표지 원칙, 콘텐츠 최신성 인지 |
| 4 | **빠른 링크** | h=80 | 카드 3개 가로 배치: 운영 FAQ / 기능 업데이트 이력 / 개선 접수, 카드: w=340, h=64, 아이콘 + 제목 | 자주 사용 섹션 1-click 접근 |

> **UX Rationale** — 홈 페이지의 '#1 히어로 검색'과 '#2 카테고리 그리드'는 각각 Known-item 검색(60%)과 Exploratory 탐색(25%)에 대응한다. 두 패턴을 동시에 제공하여 사용자의 의도에 따라 자연스럽게 분기되도록 설계한다. Notion Help Center가 정확히 이 구조를 사용하고 있으며, Zendesk Guide의 인기 테마(Flatrock, Selway)도 동일 패턴이다.

## 7. 가이드 상세 페이지 UI 설계

### 7.1 페이지 레이아웃 구조

가이드 상세 페이지는 3-column 레이아웃을 채택한다. 좌측 사이드바(260px), 본문 영역(flex, max 840px), 우측 미니맵(200px, 1440px+ 해상도에서만 노출)으로 구성된다.

### 7.2 본문 콘텐츠 블록 순서 및 UI 패턴

본문은 10개 블록을 고정 순서로 배치한다. 각 블록의 UI 패턴은 F-패턴 스캐닝과 30초 내 답 원칙에 최적화되어 있다.

| # | 블록 | UI 패턴 | 높이(px) | 인터랙션 | 레퍼런스 |
|---|---|---|---|---|---|
| 1 | **제목 + 메타 바** | H1 + 모듈뱃지 + 사용대상 + 수정일 + '업데이트됨' 뱃지 | ~80 | 정적 | 모든 서비스 공통 |
| 2 | **TL;DR 카드** | 배경색 카드, 2~3줄 요약, 좌측 Info 아이콘 | ~72 | 항상 펼침, 복사 버튼 | GitBook 요약, Notion 콜아웃 |
| 3 | **메뉴 경로** | AMS 경로 (> 구분), 우측 '바로가기' 딥링크 | h=40 | 딥링크 → AMS 이동 | Stripe 맥락 동기화 |
| 4 | **사용 방법** | 번호 스텝퍼 UI, 스텝: 제목 + 설명 + 스크린샷 | auto | 스텝 클릭 → 스크롤 | Slack 스텝 가이드 |
| 5 | **주요 항목 설명** | 2-column 테이블, 셀 내 필수/선택 뱃지 | auto | 정적 | Stripe 파라미터 테이블 |
| 6 | **운영 케이스** | 아코디언 카드, 상황 라벨 + 처리 방법 | auto | 기본 접힘, 다중 펼침 가능 | Notion 토글 블록 |
| 7 | **유의사항** | Alert 배너, 삼각형 아이콘, 노란 배경 | auto | 항상 펼침 | GitBook Warning 블록 |
| 8 | **자주 발생 문제** | 3-column 테이블 (문제/원인/해결), 행별 호버 하이라이트 | auto | 정적 | Zendesk FAQ 패턴 |
| 9 | **관련 가이드** | 가로 카드 리스트, 모듈뱃지 + 제목 + 1줄 미리보기 | auto | 클릭 → 해당 가이드 | Notion 관련 문서 |
| 10 | **문서 정보 + 피드백** | 작성자·버전·수정일, '도움됨/안됨' 투표 위젯 | h=80 | 투표 → 토스트 알림 | Zendesk 피드백 |

### 7.3 운영 케이스 아코디언 상세 설계

운영 케이스는 AMS 위키의 핵심 콘텐츠 유형이다. 상담실장이 CS 응대 중 '자신의 상황'에 해당하는 케이스를 빠르게 특정해야 하므로, 아코디언 제목에 상황 키워드를 명확히 표기하는 것이 관건이다.

| 요소 | 사양 | UX 근거 |
|---|---|---|
| 아코디언 헤더 | h=48px, padding-left=16px, 좌측: 상황 라벨 (Bold, 16px), 우측: 펼침/접힘 쉐브론 아이콘(24×24) | F-패턴 좌측 스캔에 맞춰 상황 라벨을 좌측 고정 배치 |
| 아코디언 본문 | padding=16px 24px, 처리 방법: 번호 리스트, 참고사항: 배경색 박스 | 점진적 노출: 필요한 케이스만 확장. Notion 토글 블록과 동일 패턴 |
| 기본 상태 | 모두 접힘, 키워드 매칭 시 자동 펼침 (검색 연동) | 검색으로 진입 시 해당 케이스 자동 펼침, 30초 원칙 준수 |
| 다중 펼침 | 여러 케이스 동시 펼침 허용 (뮤텍스 아코디언 사용 금지) | 케이스 간 비교가 필요한 상황 대응, 실장 세그먼트의 판단 기준 참조 시 활용 |
| 라벨 패턴 예시 | '중도 입반 시' / '퇴반 후 재입반 시' / '중복결제 발생 시' / '카드 전환결제 시' | 구체적 상황 키워드 사용, '케이스1', 'Case A' 등 추상적 라벨 금지 |

> **UX Rationale** — '운영 케이스(#6)'를 아코디언으로 설계하는 이유: 상담실장은 CS 응대 중 '자신의 상황'에 해당하는 케이스만 빠르게 찾아야 한다. 모든 케이스를 펼쳐 놓으면 스크롤이 길어져 30초 원칙을 위반한다. 아코디언 제목에 상황 키워드를 명확히 표기하여, 스캔만으로 해당 케이스를 특정할 수 있게 한다.

### 7.4 피드백 위젯 설계

| 요소 | 사양 |
|---|---|
| 질문 문구 | '이 가이드가 도움이 되었나요?' |
| 선택지 | 도움됨 (Thumbs Up) / 도움 안됨 (Thumbs Down) |
| 부정 선택 시 | 추가 입력 필드 노출: '어떤 점이 부족했나요?' (선택 사항, 200자 이내) |
| 제출 후 상태 | '의견이 반영되었습니다' 토스트 메시지, 중복 제출 방지 |
| 데이터 활용 | 스테일 콘텐츠 감지: 부정 비율 30% 이상 시 관리자 알림 |

## 8. 운영 가이드 케이스별 UX/UI 전략

AMS 위키에서 다루는 운영 가이드는 그 성격에 따라 필요한 UI 패턴이 상이하다. 가이드 유형을 6개로 분류하고, 각 유형에 최적화된 UX/UI 전략과 글로벌 레퍼런스를 정의한다.

### 8.1 가이드 유형 분류 및 UI 전략 매트릭스

| 유형 | 설명 | 대표 가이드 | 최적 UI 패턴 | 글로벌 레퍼런스 |
|---|---|---|---|---|
| **SOP 절차형 (Step-by-Step)** | 순서가 정해진 처리 절차, 실수 시 되돌리기 어려움 | 환불 처리 SOP, 계정이관 절차, 결제 전환 처리 | 번호 스텝퍼 UI, 각 스텝: 스크린샷 + 설명, 완료 체크박스(선택), 경고 포인트 인라인 배치 | Slack Help: 스크린샷 기반 스텝 가이드, 단계 번호 좌측 고정 |
| **판단 분기형 (Decision Tree)** | 상황에 따라 처리 방법 분기, 실장의 승인 판단 수반 | 환불 승인 기준, 이관 권한 판단, 입반 가능 여부 판단 | 의사결정 트리 다이어그램, IF-THEN 분기 테이블, 판단 조건 + 결과 + 사유 | Intercom: 조건별 분기 가이드, Flowchart 인라인 삽입 |
| **상황 대응형 (Case-based)** | 다양한 상황별 처리 방법, 정해진 순서 없음 | 출결 이슈 대응, 수강신청 예외 처리, 교재 배송 문제 | 아코디언 카드(상황별 접기/펼치기), 상황 라벨 + 처리 방법, 다중 펼침 허용 | Notion: 토글 블록 기반, 상황별 접기/펼치기 |
| **용어 참조형 (Reference)** | 화면 요소·설정값 설명, 자주 참조하는 레퍼런스 | AMS 화면 필드 설명, 상태값 정의, 권한 매트릭스 | 테이블(정렬·검색 가능), 필드명 + 설명 + 예시값, 뱃지: 필수/선택/자동 | Stripe Docs: 파라미터 테이블, 정렬·필터 지원 |
| **트러블슈팅형 (Troubleshoot)** | 문제 발생 시 원인 진단 → 해결 방법 제시 | 영상 재생 오류, QR 인식 실패, 결제 실패 대응 | 문제→원인→해결 3-column 테이블, 심각도 뱃지(치명/높음/보통), 관련 Jira 이슈 링크 | Zendesk Guide: FAQ 페이지 패턴, 문제/원인/해결 구조 |
| **정책 공지형 (Policy)** | 운영 정책 변경 안내, 적용일 기준 관리 | 수강료 정책 변경, 환불 기준 개정, 이관 권한 정책 | 변경 전/후 비교 테이블, 적용일 + 영향 범위 명시, 이전 버전 '접기' 표시 | Atlassian: 버전 비교 뷰, 변경 이력 diff |

#### A. SOP 절차형: 스텝퍼 UI 사양

| 스텝 컨테이너 | 좌측 번호(w=40, 원형, accent 색상), 우측 콘텐츠(flex), 스텝 간 점선 연결(좌측 번호 중앙 기준) | 스텝 클릭 시 해당 위치 스크롤, 미니맵에서 현재 스텝 하이라이트 |

## 9. 검색 UX 설계

### 9.1 검색 인터랙션 흐름 (1920×1080)

| 단계 | 사용자 행동 | UI 반응 (1920px) | 사양 |
|---|---|---|---|
| 진입 | '/' 키 또는 검색바 클릭 | 검색 오버레이 활성화 (w=640, center), 배경 반투명(#000, opacity=0.5), 최근 조회 5개 즉시 노출 | focus trap 적용, ESC 키 닫기, 300ms fade-in |
| 입력 | 검색어 타이핑 | 2글자 이상: 자동완성 결과 노출, 결과 카드: 모듈뱃지 + 가이드제목 + 스니펫 2줄, 동의어 매핑 자동 적용 | 디바운스 300ms, 최대 8개 결과, 키워드 하이라이트 |
| 선택 | 결과 클릭 또는 Enter | 가이드 상세 페이지 이동, 검색어 해당 위치 스크롤 + 하이라이트, 아코디언 자동 펼침 | ↑↓ 키보드 네비게이션, Enter 선택, 검색어 3초간 하이라이트 |
| 미발견 | 일치 결과 없음 | '결과가 없습니다' 빈 상태 + 추천 가이드 3개 + '개선/불편사항 접수' 링크 | 빈 상태 일러스트, 검색어 오타 교정 제안 |

### 9.2 탐색 모델: 하이브리드 (Navigation + Search)

운영자는 '이미 아는 것을 다시 찾는(Known-item)' 패턴과 '모르는 것을 탐색하는(Exploratory)' 패턴이 혼재한다. 이에 따라 좌측 트리 네비게이션(구조적 탐색)과 상단 글로벌 검색(즉시 접근)을 동시 제공하는 하이브리드 모델을 채택한다.

| 탐색 유형 | 사용자 의도 | UI 패턴 | 비중(추정) |
|---|---|---|---|
| Known-item 검색 | "계정이관 절차 다시 보자" | 글로벌 검색바 + 자동완성, 최근 조회 가이드 3~5개 노출 | ~60% |
| 구조적 탐색 | "결제 관련 가이드 전체 보기" | 좌측 사이드바 트리(모듈>그룹>가이드), 현재 위치 하이라이트 + 브레드크럼 | ~25% |
| 연관 탐색 | "이 가이드랑 관련된 다른 건?" | 가이드 하단 '관련 가이드' 링크, 같은 모듈 내 다음/이전 네비게이션 | ~15% |

> **UX Rationale** — 검색이 60%를 차지하므로, 검색 품질이 위키 전체 UX를 결정한다. 동의어 사전(예: '환불'='취소결제'='결제취소'), 학부모 표현→AMS 용어 매핑(예: '돈 돌려받기'→'환불'), 태그 기반 퍼지 검색이 MVP에 포함되어야 한다.

### 9.3 동의어 사전 (초기 시드: CS 1,119건 기반)

학부모/학생이 사용하는 일상 표현과 AMS 시스템 용어 사이의 괴리를 해소하기 위해, 검색 시 동의어 매핑을 적용한다. 초기 사전은 CS 데이터 1,119건에서 빈출 표현을 추출하여 구성한다.

| 사용자 표현 (검색어) | AMS 용어 (매핑 대상) | 출처 | 빈도 |
|---|---|---|---|
| 돈 돌려받기, 환불해줘, 취소결제, 결제취소 | 환불 관리 | CS 상담 113건 | 10.1% |
| 아이디 합치기, 계정 연결, 병합, 이관 | 계정이관, 통합회원 | CS 309건 | 27.6% |
| 출석 안 찍혀요, QR 안 됨, 바코드 오류 | QR 출석, 출결 관리 | CS 140건 | 12.5% |
| 영상 안 나와요, 강의 재생 안 됨, 버퍼링 | 복습영상 재생 오류 | CS 258건 | 23.1% |
| 수업 옮기기, 반 바꾸기, 전반 | 입반/퇴반 관리 | CS 추정 | - |
| 결제 두 번 됐어요, 이중결제, 중복 빠짐 | 중복결제 처리 | CS 상담 | - |

> **Reference** — UXCC(2025)는 지원 콘텐츠의 검색성(Findability)을 강조하며, '사용자가 사용하는 언어로 제목과 헤딩을 작성하라'고 권고한다. 내부 용어를 제목에 사용하면 검색 실패율이 급증한다. 동의어 사전은 정적 JSON이 아니라, FAQ 피드백과 '검색 미발견' 로그를 기반으로 주기적(월 1회)으로 갱신해야 한다. 4단계 AI 챗봇 구축 시 자동 동의어 확장 파이프라인으로 전환한다.

## 10. 가이드 작성 툴 UI 상세 설계

### 10.1 설계 목적 및 방향

가이드 작성 툴은 관리자(기획자)가 표준화된 형식으로 가이드를 입력·발행하는 내부 도구이다. '문서를 직접 작성'하는 것이 아니라 '정해진 항목에 내용을 채우는' 구조화된 입력 방식을 채택한다.

| 설계 원칙 | 설명 |
|---|---|
| 문서 계층 구조 선행 입력 | 모듈/그룹/가이드명/세부구분을 먼저 정의한 후 본문 작성 |
| 공통 블록 기준 입력 | 본문은 기능설명, 사용방법, 운영케이스 등 표준 블록으로 구성 |
| HTML 템플릿 자동 적용 | 저장 시 공통 HTML 템플릿이 자동 적용되어 결과물 생성 |
| 경로 기반 관리 | 같은 제목 허용, Path로 구분 관리 |

### 10.2 작성 툴 레이아웃 (1920×1080)

3-패널 구성: **좌측 타입 템플릿 picker** + **상단 액션 바** + **본문 편집/미리보기 토글**. 미리보기는 별도 패널이 아니라 Tab 토글로 전환되어 편집 영역의 가독성을 우선한다.

| 영역 | 너비(px) | 콘텐츠 | 인터랙션 |
|---|---|---|---|

### 10.2.1 좌측 템플릿 picker 동작

| Step | 동작 | 결과 |
|---|---|---|
| 1 | 좌측 6개 카드 중 하나 선택 (예: TROUBLE) | `meta.type` = 'TROUBLE', 선택 카드 dark 강조, 본문 content는 `EMPTY_CONTENT` 로 초기화 |
| 2 | 본문 탭에서 자동 노출되는 sections 확인 | `SECTIONS_BY_TYPE[meta.type]` 에 따라 주의사항 / troubleTable / cases 자동 구성 |
| 3 | 메타 탭에서 유형 Select로 변경 | sections 자동 재구성 (좌측 picker와 동일 효과) |

> 좌측 picker는 "어떤 가이드를 만들지 처음 고르는 단계"이므로 검색·필터를 두지 않는다. 6개 옵션은 모두 한 화면에 노출된다.

## 11. 저장·발행·버전 관리 정책

### 11.1 상태 전이 다이어그램

| 현재 상태 | 액션 | 다음 상태 | 권한 | UI 트리거 |
|---|---|---|---|---|
| (없음) | 새 가이드 작성 시작 | 작성중 (Draft) | 관리자 | '새 가이드' 버튼 |
| 작성중 | 검수 요청 | 검수중 (Review) | 관리자 | '검수 요청' 버튼 + 리뷰어 선택 |
| 검수중 | 수정 필요 판단 | 작성중 (Draft) | 리뷰어 | '반려' 버튼 + 코멘트 필수 |
| 검수중 | 승인 및 발행 | 배포완료 (Published) | 리뷰어 | '승인·발행' 버튼 + 최종 확인 모달 |
| 배포완료 | 수정 발행 (업데이트) | 배포완료 (v+1) | 관리자 | '수정' 버튼 → 편집 모드 → 재발행 |

> **UX Rationale** — '검수중' 상태를 추가한 이유: 원본 기획서에는 작성중/배포완료 2단계만 정의되어 있으나, 잘못된 SOP가 발행될 경우 운영 오류로 직결된다. 리뷰어 검수 단계를 거쳐야 콘텐츠 신뢰성이 담보된다.

### 11.2 최신 버전 단일 노출 정책

사용자는 항상 최신 버전 1개만 조회 가능하다. 이전 버전은 히스토리로만 관리하며 UI에는 노출하지 않는다.

| 정책 | 사양 | UX 처리 |
|---|---|---|
| 단일 노출 | 사용자: 항상 active 버전만 조회, 관리자: 히스토리 뷰에서 전 버전 접근 | 관리자 전용 '버전 이력' 드로어, 버전 간 diff 비교 뷰 |
| URL 고정 | 버전 업데이트 시 URL 불변, 항상 최신 내용으로 렌더링 | OG 메타태그 동적 갱신, 공유 링크 프리뷰 최신 제목 반영 |
| 롤백 | 관리자가 이전 버전을 active로 전환 가능, 현재 버전은 inactive 처리 | '되돌리기' 버튼 + 확인 모달, 롤백 사유 입력 필수 |
| 업데이트 표시 | 최근 업데이트 일자 + '업데이트됨' 뱃지, 가이드 목록에서도 뱃지 노출 | 뱃지: 발행 후 7일간 노출, 날짜: 항상 표시 |

### 11.3 내부 히스토리 관리

사용자에게는 비노출되지만, 내부적으로 모든 버전 이력을 유지하여 필요 시 rollback이 가능하다.

| 버전 | 수정일 | 작성자 | 상태 |
|---|---|---|---|
| v0.1 | 2026-03-01 | 담당자 A | inactive |
| v0.2 | 2026-03-10 | 담당자 B | active |

### 11.4 결과물 출력 구조

입력 완료 후 HTML은 다음 순서로 출력된다: 가이드 제목 → 기능 설명 → 메뉴 경로 / 사용 대상 → 사용 방법 → 주요 항목 설명 → 운영 케이스 → 유의사항 → 자주 발생하는 문제 → 관련 기능 → 작성자 / 수정일 / 버전

| 항목 | 입력 예시 |
|---|---|
| 모듈 (Depth 1) | 청구 |
| 가이드 그룹 (Depth 2) | 청구 가이드 |
| 가이드명 (Depth 3) | 청구 목록 확인하기 |
| 세부 구분 (Depth 4) | 회원상세 > 청구 목록 확인 |

| 가이드 본문 블록 | 위키 전용 정의 | TldrCard, StepperBlock, CaseAccordion, AlertBanner, FeedbackWidget |
| 가이드 작성 툴 | HDS 공통 + 위키 확장 | Select, TextInput, Checkbox (HDS) + StepEditor, CaseCardEditor (위키 확장) |

| StaleAlert | 90일 미수정 알림 | guides[], onDismiss | 노출/해제 | Alert 확장 |

### 13.3 접근성 기준 (WCAG 2.1 AA)

| 기준 | 사양 | 적용 대상 |
|---|---|---|
| 키보드 네비게이션 | Tab 순서: 검색바 → 사이드바 → 본문 → 미니맵, 모든 인터랙티브 요소에 :focus-visible 스타일 적용 (2px solid #2E75B6) | 전체 페이지 |

### 13.4 모바일 대응 전략

상담실장이 학원 현장에서 모바일(주로 iOS Safari)로 위키에 접근하는 시나리오가 존재한다. 별도 네이티브 앱은 구축하지 않으며, 반응형 웹으로 대응한다.

| 요소 | 데스크톱 (≥1024px) | 모바일 (<1024px) |
|---|---|---|
| 사이드바 | 260px 고정 노출 | 하단 시트(Bottom Sheet) 전환 |
| 검색 | 상단 검색바 고정 | 상단 검색 아이콘 → 풀스크린 검색 |
| 본문 | max 840px 중앙 정렬 | 풀 너비, 16px 좌우 패딩 |
| 미니맵 | 200px 우측 고정 | 비노출 |
| 운영 케이스 아코디언 | 기본 접힘 | 기본 접힘 (터치 영역 48px 이상) |
| 가이드 작성 툴 | 풀 기능 지원 | 조회 전용 (작성·편집 비활성화) |

> **UX Rationale** — 모바일에서 '가이드 작성 툴'을 조회 전용으로 제한하는 이유: 복잡한 폼 입력(스텝 순서 변경, 테이블 편집)은 터치 환경에서 오류 가능성이 높다. 작성·편집은 데스크톱 전용으로 한정하고, 모바일에서는 '조회·검색·공유'에 집중한다.

### 14.1 분석 데이터 개요

AMS 위키 콘텐츠 우선순위 도출을 위해 마이클래스 카카오톡 상담 채널(고객 대면)과 단과 상담실장 오픈채팅(내부 운영) 데이터를 교차 분석하였다.

| 단과_kakao_chats | 상담실장 53명 내부 운영 채팅 126건, 8개 이슈 카테고리 분류 | 7일 (2026.03.04~03.10) |

### 14.2 우선순위 매트릭스

위키 콘텐츠 우선순위는 '상담 빈도 × 부정감정 비율 × 응답시간'의 복합 지표로 결정하였다.

| 등급 | 위키 섹션 | 근거 데이터 | 가이드 유형 | 비중 |
|---|---|---|---|---|
| **P0** | 회원관리 > 통합회원/계정이관 | 고객 309건(27.6%) + 직원 24건(19.0%) | SOP 절차형 | 27.6% |
| **P0** | 영상/VOD > 복습영상 접근·재생 오류 | 고객 258건(23.1%), 부정 17.4% | 트러블슈팅형 | 23.1% |

| **P2** | 교재/배송 | 고객 19건 | SOP 절차형 | 1.7% |
