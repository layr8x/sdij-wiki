# 중복 계정 통합 프로세스

> Confluence FVSOL/1614381124 · 추출 2026-05-20

## AMS 데이터 정의
상담, 접수, 입반, 결제 등 회원의 이력 및 결과 데이터.

## 케이스 매트릭스

### 로컬 2개

| 케이스 | 결과 | 처리 방법 |
|---|---|---|
| 1개만 AMS 데이터 있음 | 데이터 있는 쪽 유지 | 없애는 계정 핸드폰 번호 010-0000-0000 변경 + '미사용' 태그 처리 |
| 둘 다 AMS 데이터 있음 | 한 쪽 미사용 처리 | 케이스별 분기 (아래) |

### 둘 다 데이터 있음 — 세부 케이스

**미사용에 결제 있는 경우:**
1. 없애는 계정에서 결제취소
2. 사용할 계정에서 재결제 안내
3. 재결제 확인 후 없애는 계정은 010-0000-0000 변경 + '미사용' 태그

**미사용에 입반/접수 있는 경우:**
1. 없애는 계정에서 퇴반/접수취소
2. 사용할 계정에서 입반/접수완료
3. 없애는 계정은 010-0000-0000 변경 + '미사용' 태그

**단, 미사용할 대기 있음:**
- 대기순번 유지하며 이관 불가
- 학생 의사 확인 후 안내:
  - 순번 유지가 더 중요: 기존 계정 통해 확인
  - 계정 이관이 더 중요: 기존 계정에서 대기접수 취소 후 새 계정에서 신규로 대기접수 (순번은 가장 후순위로 변경)

### 통합 1개 + 로컬 1개

| 케이스 | 결과 | 처리 |
|---|---|---|
| 통합에 AMS 데이터 없음 | 수동 처리 사항 없음 | 회원이 직접 통합회원 로그인 후 계정 연동 → 통합회원으로 회원 데이터 병합 |
| 통합에 AMS 데이터 있음 | 기능으로 해소 불가 | 개발요청 필요 (Jira SERP-5958) |

## 개발 요청 (SERP-5958)
해당 학생(계정)과 관련된 모든 이력 데이터를 하나하나 수작업으로 이관. 완료 일정은 안내 어려움.
