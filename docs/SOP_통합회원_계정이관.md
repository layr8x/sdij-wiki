## 1. 도메인 모델 — 시대인재 계정 5종

| 계정 종류 | 생성 방식 | 인증 | 용도 |
|---|---|---|---|
| **임시계정** | 학원 모집 시 시스템 자동 생성 (이름+전화번호) | 없음 | 학원 수강 등록 holder |
| **통합회원 (시대인재 C)** | 사용자가 myclass.sdij.com 에서 직접 가입 | 이메일 인증 | 시대인재 통합 ID |
| **마이클래스 계정** | 마이클래스 앱 회원가입 | 이메일 + 학생 휴대폰 등록 | 인강·VOD 수강 |
| **라이브 계정** | 라이브 수업 등록 | **PASS 본인인증** | 현강 라이브 |
| **북스 계정** | 시대인재북스 사이트 별도 가입 | 이메일 | 교재 구매 |

### 핵심 제약 사항
- ⚠️ **본인인증(PASS) 계정은 1건만 사용 가능** — 동일 휴대폰 번호로 여러 계정 등록 불가
- ⚠️ **라이브 → 마이클래스 이관은 내부 정책상 불가** (역방향만 가능)
- ⚠️ 동일인이 임시계정 + 통합회원 둘 다 활성 상태면 **자동 연결 실패** (중복 충돌)

## 2. 케이스 분기 — 5개 시나리오

### CASE A: 임시계정 ↔ 통합회원 자동 연결 실패 (가장 흔함)

**증상**: 사용자가 통합회원 가입 후 마이클래스 앱 로그인 시 수강정보 안 보임

**진단**: 
1. 임시계정에 수강정보 등록되어 있음
2. 사용자가 통합회원 직접 가입했지만 동일인의 다른 계정과 중복으로 자동 연결 충돌

**해결책 (사용자 안내)**:
> 옵션 1 — 사용자 직접 (즉시 가능)
> ① myclass.sdij.com 접속 → 해당 계정으로 로그인 → 회원탈퇴
> ② 다시 회원가입 → 웹 내 자동연결 기능으로 해결
> 
> 옵션 2 — 운영 측 수동 이관 (1-2일 소요)
> 수강정보 이관 대상 계정(이메일)을 본 채널로 전달 → 개발팀에 요청

---

### CASE B: 학생 휴대폰 번호 PASS 충돌

**증상**: "마이클래스 계정에 학생 휴대폰 번호 등록이 안 돼요"

**진단**: 학생 휴대폰 번호가 다른 라이브 계정에 PASS 본인인증으로 이미 등록됨

**해결책 (사용자 안내)**:
> 본인인증 계정은 1건만 사용 가능합니다. 아래 2가지 중 선택해주세요.
> 
> ① **계정 2개 병행 사용**: 마이클래스 계정 + 라이브 계정 각각 사용 (PASS 등록은 라이브에 유지)
> 
> ② **이관 처리**: 마이클래스 데이터를 라이브 계정으로 이관 → PASS 인증된 라이브 계정 1개로 통합 사용

---

### CASE C: 이메일 계정 2개 (gmail / naver 등) — 데이터 분산

**증상**: "계정이 두 개인데 어느 게 진짜 데이터인지 모르겠어요"

**진단**: 사용자가 모르고 두 번 가입. 학생 휴대폰 번호는 한 계정에만 등록 가능.

**해결책 (사용자 안내)**:
> 확인 결과 두 계정이 있습니다.
>   - {gmail_id} (학생 번호 미등록)
>   - {naver_id} (학생 번호 등록됨)
> 
> 학생 번호 등록된 {naver_id} 계정으로 데이터를 모두 **이관 완료**해두었습니다.
> 안심하시고 {gmail_id} 계정은 직접 회원탈퇴 진행해주시면 됩니다.
> 탈퇴 후 {naver_id} 계정에서 학생 번호 재등록하시면 정상 이용 가능합니다.

**선결 조건**: 운영 측에서 두 계정 식별 → 학생 번호 등록된 계정 기준으로 이관 → 사용자에게 통보

---

### CASE D: 마이클래스 ↔ 라이브 ↔ 북스 3개 계정 얽힘

**증상**: "수강신청은 라이브에서 했는데 마이클래스에서 영상이 안 나와요"

**진단**: 마이클래스/라이브/북스 별도 시스템. 정책상 라이브→마이클래스 이관 불가.

**해결책 (사용자 안내)**:
> 내부 정책상 라이브 계정을 마이클래스 계정으로 이관하는 것은 어렵습니다.
> 또한 학생 휴대폰 번호는 PASS 본인인증으로만 등록 가능하여, 고객센터에서 임의 등록 불가합니다.
> 
> 다만, 아래와 같이 진행 가능합니다.
> 
> ① 마이클래스 계정 ↔ 라이브 계정 **이메일 통합**: 동일 이메일로 양쪽 가입 시 마이클래스 → 라이브 데이터 이관 처리
> ② 마이클래스 계정 회원 탈퇴 후, 라이브 계정에 학생 휴대폰 번호 등록

---

### CASE E: 동일인 중복 등록으로 시스템 충돌

**증상**: 통합회원에도 학생 등록, 임시계정에도 동일 학생 다른 수업 등록 → 자동 연결 실패

**진단**: 두 계정에 동일 이름의 학생이 동시에 활성. 시스템이 어느 쪽이 진짜인지 결정 못함.

**해결책 (운영 측 처리)**:
1. **퇴원 정리** — 한쪽 계정의 수업을 퇴원 처리 (보통 수업 적은 쪽)
2. 옵션 ① 사용자 측 회원탈퇴 후 재가입 (즉시, 번거로움)
3. 옵션 ② 개발팀 데이터 재이관 (+1일 소요, 사용자 대기)

⚠️ **퇴원 이력이 있으면 자동 연결 안 됨** — 옵션 ② 권장

## 3. 표준 응답 템플릿 (Confluence·Slack·카톡 모두 사용)

### 템플릿 T-001: 임시계정·통합회원 안내 (최빈 응답)

```
회원마다 본인의 인증정보로 가입된 통합회원 계정은 시스템에서 각각 다른 개인으로 관리됩니다.

최초 모집 시 이름/전화번호만으로 학원에 수강 등록되어있는 임시 계정이 발생하고,
회원분들께서 직접 가입하신 통합회원으로 임시 계정의 수강정보 연동을 직접 하시도록 안내하고 있습니다.

이미 학원의 모든 수강정보는 [기존_이메일] 계정의 소유자인 개인에게 등록이 되어있음을 안내드립니다.

회원님께서 원하신다면 데이터를 관리하는 개발팀에서 새로운 계정으로 입반, 결제, 출석 등의 정보를 
하나씩 수동 이관해 드릴 수 있습니다.

다만 이 작업은 1~2일 정도 소요될 수 있는 점 양해 부탁드리며,
그렇게 진행하시길 원한다면 수강정보를 이관할 대상 계정을 이 채널로 전달해주시기 바랍니다.
```

### 템플릿 T-002: PASS 본인인증 충돌 (CASE B)

```
라이브에서 수강 중인 계정에 학생 휴대폰 번호가 PASS 본인인증으로 이미 등록되어 있어,
본인인증 계정은 1건만 사용 가능함에 따라 마이클래스 계정에 동일 번호 등록이 어렵습니다.

해결 방법은 아래 2가지입니다.
  1. 마이클래스 계정과 라이브 계정을 분리하여, 2개 계정으로 사용
  2. 마이클래스 계정을 라이브 계정으로 이관 처리하여, 본인인증으로 가입된 라이브 계정 1개 사용
```

### 템플릿 T-003: 이메일 2개 데이터 이관 완료 (CASE C)

```
확인 결과, 아래와 같이 두 개의 계정이 등록되어 있습니다.
  [email_1] (학생 번호 미등록)
  [email_2] (학생 번호 등록됨)

학생 번호는 동일 번호를 여러 계정에서 중복 사용하실 수 없어
현재 [email_1] 계정에서는 학생 번호 등록이 불가능한 상태입니다.

이에 따라 [email_1] 계정에서 사용하시던 데이터를 모두 [email_2] 계정으로 이관 완료해두었으니
안심하시고 [email_1] 계정은 회원 탈퇴만 직접 진행해주시면 됩니다.

탈퇴 후 [email_2] 계정에서 학생 번호를 다시 등록해주시면 정상 이용 가능합니다.
```

## 4. 운영자 처리 체크리스트

### Phase 1 — 진단 (5분)
- [ ] **사용자 이메일·전화 확보** (양식: 회원명·회원ID·연락처·문의내용)
- [ ] AMS 백오피스에서 동명·동전화 계정 검색
- [ ] 결과 확인:
  - 임시계정만 있음 → CASE A
  - 마이클래스 + 라이브 둘 다 있음 → CASE B 또는 D
  - 이메일 도메인 다른 계정 2개 → CASE C
  - 통합회원 + 임시 둘 다 활성 → CASE E

### Phase 2 — 응답 (5분)
- [ ] 해당 케이스 템플릿 (T-001~T-004) 선택해서 응답
- [ ] 변수 치환 (사용자 실제 이메일·번호)
- [ ] 사용자에게 옵션 선택 요청 (직접 처리 vs 이관 의뢰)

### Phase 3 — 이관 의뢰 (사용자 옵션 ② 선택 시)
- [ ] 단과 운영 톡방에 이관 요청 메시지 (양식: from / to / 사유)
- [ ] 플서실 황인규 QA 또는 차주희 QA 처리
- [ ] 1-2일 후 이관 완료 통보 받으면 사용자에게 후속 안내

### Phase 4 — 추적
- [ ] CS 시스템에 케이스 분류 태그 (CASE A/B/C/D/E)
- [ ] 재문의율 추적 (1주일 내 같은 사용자 재문의 발생 시 SOP 보완 검토)

## 8. 데이터 출처

- 매니저 raw 응답 80건 (`kakao_partner_messages` 황*규·차*희·신*현 sender_id 필터)
- 사용자 첫 메시지 30건 (통합·이관·중복 키워드)
- 시스템 자동 안내 5건 (양식 안내·재로그인 안내)

근거 SQL:
```sql
select message from kakao_partner_messages
where sender_type='manager'
  and message ~* '(통합|이관|로컬 미사용|중복가입|이중계정)'
order by length(message) desc;
```
